# gitops/ — per-company cluster layout

## Topology

Each company gets one k3s node (provisioned by E1.1) running:

```
flux-system          # Flux controllers + sync config
ingress-nginx        # Single ingress controller; wildcard TLS cert pre-mounted
cert-manager         # For future per-company certs; not on the demo critical path
<app namespaces>     # One namespace per app (keycloak, nextcloud, mail, …)
```

All resources are owned and reconciled by **Flux v2**. No manual `kubectl apply`
after bootstrap — everything goes through a GitOps PR.

## How a new company cluster is created

1. `infra/stacks/company/` provisions the VM (E1.1) → emits `node.public_ip`.
2. `scripts/bootstrap-cluster.sh` SSHes to the node, installs Flux, uploads the
   wildcard TLS secret, and applies the rendered cluster manifests.
3. Flux reconciles `gitops/clusters/<company_id>/` — infra layer first, then apps.
4. New subdomains (`auth.<domain>`, `files.<domain>`, …) are instantly reachable
   because the wildcard cert was pre-mounted before reconciliation started.

## Adding a new company's cluster directory

The deploy-trigger renders `gitops/clusters/company-template/` with substitutions
and applies it directly. A permanent per-company directory is committed to the repo
so Flux can reconcile it on future runs:

```
gitops/clusters/
├── company-template/   # Template — never deployed directly
└── acme-demo/          # Generated by bootstrap-cluster.sh and committed
    ├── kustomization.yaml
    ├── flux-system/
    ├── infra/
    └── apps/
```

## Layering order

```
flux-system/ → infra/ → apps/
```

Flux applies layers in dependency order. Apps only start after ingress and
cert-manager are healthy.

## Adding a new app

See `charts/README.md` for the chart convention, then:
1. Add a `HelmRelease` YAML to `gitops/clusters/company-template/apps/`.
2. Add it to `apps/kustomization.yaml`.
3. PR merged → Flux reconciles all existing company clusters automatically.
